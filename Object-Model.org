* Perseus6 Object Model — Design Document

#+date: 2026-04-12
#+author: Clifford Wulfman

This document describes the object model for the Perseus6 MVP build
pipeline.  It is a living document; revisions should be committed to
the wiki as the design evolves.

** Design Principles

*** Command pattern, not function pattern

Perseus6's compiler objects are /agents/ that produce artifacts; they
are not functions that transform inputs into outputs.  This distinction
is deliberate and should be maintained throughout the implementation.

A compiler receives a document and an output location, writes its
artifacts to disk, and returns =None= on success or raises an exception
on failure.  The return value is confirmation; the artifacts are the
output.  This is analogous to a Makefile recipe: success is evidenced
by the target file on disk, not by a return value.

This design has an important practical consequence: pipeline stages are
/independently restartable/.  If the catalog compilation fails, the
chunk HTML files already on disk do not need to be regenerated.  Each
stage can inspect the disk to determine what work remains.

*** Disk as intermediate state

Data flows between pipeline stages via the filesystem, not via in-memory
data structures.  The XSLT chunking pipeline writes =index.json= manifest
files to disk; later pipeline stages read those files back in when they
need to reason about chunk structure.

This means pipeline stages are loosely coupled: each stage depends on
the /artifacts/ of prior stages, not on their in-memory objects.  A
future Make-based build system would be a natural fit for this architecture.

*** Separation of concerns

- =TEIDocument= knows how to parse itself and expose its content.
- =TEIMetadata= carries descriptive information about a document without
  holding the document tree.
- =ChunkingStrategy= knows how to segment a document; compilers do not.
- =SiteMap= owns the URL/path scheme; nothing else constructs output paths.
- =BuildPipeline= orchestrates; it does not implement compilation logic.

** Object Model

*** Data Objects

Data objects are plain dataclasses: no significant behavior beyond
field access and construction.

#+begin_src python
@dataclass
class TEIMetadata:
    urn: str
    title: str
    author: str
    language: str       # BCP 47 / ISO 639: 'grc', 'lat', 'eng', etc.
    text_type: str      # 'verse' | 'prose' | 'drama'
    chunk_unit: str     # 'card' | 'section' | 'line' | etc.
    source_path: Path


@dataclass
class ChunkManifestEntry:
    n: str              # chunk identifier (milestone/@n or div/@n)
    file: str           # relative path of the compiled HTML file
    urn: str            # full CTS URN range for this chunk


@dataclass
class ChunkManifest:
    base_urn: str
    title: str
    chunks: list[ChunkManifestEntry]
#+end_src

=ChunkManifest= is the Python representation of the =index.json= file
produced by the XSLT chunking pipeline.  It is not passed between
pipeline stages in memory; each stage that needs it deserializes it
from disk.

*** Errors

#+begin_src python
@dataclass
class CompilationError(Exception):
    document: TEIDocument
    message: str
    cause: Exception | None = None
#+end_src

Compilers raise =CompilationError= on failure.  The =BuildPipeline= may
adopt a fail-fast or collect-all-errors policy; the latter is preferred
for batch builds over large corpora.

*** Document Layer

=TEIDocument= parses and holds a single TEI source file.  Metadata
extraction is eager: the =metadata= property is populated on
construction.  If performance over the full corpus (currently ~3,600
documents) proves problematic, lazy extraction can be introduced later.

#+begin_src python
class TEIDocument:
    def __init__(self, path: Path) -> None: ...

    @property
    def tree(self) -> etree._ElementTree: ...

    @property
    def metadata(self) -> TEIMetadata: ...

    @classmethod
    def from_path(cls, path: Path) -> TEIDocument: ...
#+end_src

*** Corpus Layer

=Corpus= discovers and enumerates TEI source documents under a root
directory.  It does not load documents eagerly; =documents()= is a
lazy iterator.

#+begin_src python
class Corpus:
    def __init__(self, root: Path) -> None: ...

    def documents(self) -> Iterator[TEIDocument]: ...
    def document(self, urn: str) -> TEIDocument: ...
#+end_src

*** Chunking Strategy Layer

A =ChunkingStrategy= encapsulates the logic for determining how a
document should be divided into navigable chunks.  Compilers receive a
strategy as a collaborator; they do not contain chunking logic
themselves.

Strategy selection is handled by a =StrategySelector= collaborator
rather than by =TEIDocument= itself, keeping document parsing and
segmentation logic separate.  The selector reads the =refsDecl= as a
hint, but also inspects the actual document structure, since encoding
inconsistencies in the Perseus corpus mean that =refsDecl= declarations
cannot always be trusted to match reality.

Strategies may be overridden by explicit per-document or
per-collection configuration, with automatic detection as a fallback.

#+begin_src python
class ChunkingStrategy(ABC):
    @abstractmethod
    def describes(self, doc: TEIDocument) -> bool: ...


class MilestoneStrategy(ChunkingStrategy):
    """Chunk at <milestone unit='{unit}'/> elements."""
    def __init__(self, unit: str) -> None: ...


class DivisionStrategy(ChunkingStrategy):
    """Chunk at <div type='{type}'> elements."""
    def __init__(self, type: str) -> None: ...


class StrategySelector:
    def select(self, doc: TEIDocument) -> ChunkingStrategy: ...
#+end_src

*** Site Map Layer

=SiteMap= owns the output path and URL scheme for all compiled
artifacts.  No other object constructs output paths; everything goes
through =SiteMap=.  This means the URL scheme can be changed in one
place.

#+begin_src python
class SiteMap:
    def __init__(self, output_root: Path) -> None: ...

    def chunk_dir(self, urn: str) -> Path: ...
    def chunk_path(self, urn: str, chunk_id: str) -> Path: ...
    def manifest_path(self, urn: str) -> Path: ...
    def catalog_path(self, language: str) -> Path: ...
#+end_src

*** Compiler Layer

Compilers follow the command pattern: =compile()= returns =None= on
success and raises =CompilationError= on failure.  All output is
written to paths obtained from =SiteMap=.

#+begin_src python
class PageCompiler:
    """Compiles a TEIDocument into a sequence of HTML chunk pages.

    Delegates document segmentation to a ChunkingStrategy and
    HTML generation to an XSLT stylesheet via Saxon.
    """
    def __init__(self, strategy: ChunkingStrategy,
                 xslt_path: Path) -> None: ...

    def compile(self, doc: TEIDocument,
                output_path: Path) -> None: ...


class CatalogCompiler:
    """Compiles a collection of TEIMetadata records into catalog HTML pages."""
    def __init__(self, site_map: SiteMap) -> None: ...

    def compile(self, entries: list[TEIMetadata],
                output_path: Path) -> None: ...
#+end_src

*** Orchestration Layer

=BuildPipeline= drives the full Milestone 1 build: iterating over the
corpus, compiling chunk pages, collecting metadata, and compiling the
catalog.  It owns the error-handling policy.

#+begin_src python
class BuildPipeline:
    def __init__(self, corpus: Corpus, site_map: SiteMap,
                 output_root: Path) -> None: ...

    def run(self) -> None: ...
#+end_src

** Object Interaction — Milestone 1 Build

#+begin_example
BuildPipeline.run()
  │
  ├── Corpus.documents()          # lazy iteration over TEI files
  │     └── TEIDocument.from_path()
  │           └── TEIMetadata     # eager extraction on construction
  │
  ├── StrategySelector.select()   # per document
  │
  ├── PageCompiler.compile()      # per document
  │     ├── [XSLT via Saxon]
  │     └── writes: chunk_1.html … chunk_N.html, index.json
  │
  └── CatalogCompiler.compile()   # once, after all documents
        ├── reads: list[TEIMetadata] collected above
        └── writes: catalog HTML pages
#+end_example

** Open Questions

1. *=SiteMap= and URN structure.* The output path scheme for chunk
   files needs to be decided before Milestone 1 is complete, since
   later milestones depend on stable URLs.  A natural scheme is
   =/{language}/{textgroup}/{work}/{version}/= but this should be
   confirmed against the CTS URN structure of the actual corpus.

2. *=StrategySelector= configuration.* Should per-document strategy
   overrides live in a configuration file checked into the repository,
   or should they be derivable from the corpus metadata alone?

3. *Corpus scope.* Currently only =canonical-greekLit= is a submodule.
   =canonical-latinLit=, =canonical-engLit=, and =canonical-pdlrefwk=
   will need to be added.  =Corpus= should support multiple root
   directories or a collection of submodule roots.

** Deferred

- =ChunkingStrategy= for hierarchical division-based texts (the
  Sophocles case): design deferred until the corpus analysis is
  complete.
- =refsDecl= parsing as a strategy-selection signal: noted but not
  yet designed; see separate discussion in planning notes.
- Makefile-based incremental builds: the disk-as-intermediate-state
  design is compatible with this but it is not being implemented now.
